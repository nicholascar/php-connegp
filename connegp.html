ConnegP for the Organisations Register

Content Negotiation by Profile for the AGLDWG's Organisations Register

This page describes how to access the content of the Australian Government Linked Data Working Group's Organisations Register, which is a list of organisations of relevance to the Working Group, in a number of different formats and according to multiple that is data specifications or profiles of them. The methods providing access to these multiple formats and profiles follow the W3C's functional specification of Content Negotiation by Profile.

Organisations Register

The Organisations Register is just a list of organisation and some metadata about them, but there is more than one way of presenting this information; different data specifications or profiles of them relevant to lists of things have different requirements around which properties of the individual things within lists, and properties of the list of them as a whole, need to be presented.

The default profile, conformed to by representations of this Organisations Register is the Registry Ontology. That specification requires items within a register – a listing that indicates some sort of authority's acceptance of things within the lists – are identified by a URI and indicate their status within the register with status terms taken from the Registry Status Vocabulary.

Also supported by the system that delivers this Organisations Register's content is a very plain "URI List" profile that just lists the organisation URIs without statuses or elven names (labels) in a simple text format. This very simple representation is useful if you just want the IDs (URIs) of things in the register.

Using ConnegP to get content

Content Negotiation by Profile (ConnegP) describes several ways in which different representations of a resource may be requested. Functional Profiles of ConnegP describes how to perform requests in different environments, such as machine-to-machine HTTP or "human user" web browser.

Get Resource By Profile

According to ConnegP's HTTP Functional Profile, an Internet client sets an Accept-Profile header to indicate the specific representation of a resource it prefers. The values it uses for the header are URIs of profiles or data specifications. For example, the simple "URI List" profile view of the Organisations Register can be retrieved using the command line tool cURL, available on many platforms, that is able to set HTTP headers. The URI for the "URI List" profile is https://w3id.org/profile/uri-list so the cull cURL command is:

~$ curl --request 'Accept-Profile: <https://w3id.org/profile/uri-list>' http://www.linked.data.gov.au/org/

Where the URI identifying the profile the resource representation conforms to is enclosed in '<' & '>' and the URI of the resource, the Register, is http://linked.data.gov.au/org/

The result you should get will be something like this: 

http://linked.data.gov.au/org/abs
http://linked.data.gov.au/org/ausgin
http://linked.data.gov.au/org/agldwg
http://linked.data.gov.au/org/anu
http://linked.data.gov.au/org/ardc
http://linked.data.gov.au/org/ato
...
http://linked.data.gov.au/org/psma
http://linked.data.gov.au/org/ror
http://linked.data.gov.au/org/tern

This command is the get resource by profile function of ConnegP implemented in HTTP. see the specification for all the possibilities available for HTTP get resource by profile requests, including preference ordering etc.

List Profiles

To know what representations of a resource are available, systems that implement ConnegP for HTTP list them all for you in an HTTP Link header. This system makes available its content according to the following profiles:

  1. Registry Ontology – http://purl.org/linked-data/registry
  2. URI List - https://w3id.org/profile/uri-list

And, mostly for demonstration, it makes available a representation that lists these representations in RDF formats:

  1. Alternate Representations Data Model - http://www.w3.org/ns/dx/conneg/altr

The Link header listing all profiles also lists all the Media Types (formats) that each profile is available in, as per the HTTP header implementation of the Alternate Representations Data Model in the ConnegP specification. The header is returned for every representation of a resource so if you just load the default profile and Media Type of the Organisations Register, which you get by just visiting http://linked.data.gov.au/org/ in a browser and which are Registry Ontology (profile) and HTML (Media Type) respectively, you will see this in the HTTP header: 

<http://linked.data.gov.au/org/ rel="self"; type="text/html"; profile="http://purl.org/linked-data/registry",
<http://linked.data.gov.au/org/ rel="alternate"; type="text/turtle"; profile="http://purl.org/linked-data/registry",
<http://linked.data.gov.au/org/ rel="alternate"; type="text/uri-list"; profile="https://w3id.org/profile/uri-list",
<http://linked.data.gov.au/org/ rel="alternate"; type="text/html"; profile="http://www.w3.org/ns/dx/conneg/altr",
<http://linked.data.gov.au/org/ rel="alternate"; type="text/turtle"; profile="http://www.w3.org/ns/dx/conneg/altr"

NOTE: HTTP Link headers do not have linebreaks. This example is line-broken to make it easier to read.

So, looking at the content of the Link header above, you can see that the representations of the Organisations Register conforming to the profile identified by https://w3id.org/profile/uri-list, the "URI List" profile, are only available in a single Media Type (format) of text/uri-list but that the representations conforming to the Registry Ontology profile are available in text/turtle & text/html which are the machine- and human-readable versions of that profile with the latter being the default view, as indicated by the rel="self" key/value pair.

Using the List Profiles response above and general knowledge of HTTP requests, Internet clients (people or machines) can formulate get resource by profile requests for the Organisations Register such as these:

  • curl --header 'Accept-Profile: <http://purl.org/linked-data/registry>' --header 'Accept: text/turtle' http://linked.data.gov.au/org/
    • The RDF (turtle) format of the Registry Ontology profile
  • curl --header 'Accept-Profile: <http://purl.org/linked-data/registry>' http://linked.data.gov.au/org/
    • The default format (HTML) the Registry Ontology profile
      • , which is the default profile
  • curl --header 'Accept-Profile: <http://www.w3.org/ns/dx/conneg/altr>' --header 'Accept: text/turtle' http://linked.data.gov.au/org/
    • The RDF (turtle) format of the Alternate Representations Data Model profile
